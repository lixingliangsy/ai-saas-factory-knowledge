# 阶段 06 · MVP 实验设计（草案）

> 项目：一人公司出海 · 创始人 李星亮
> 主验产品：scifigure-pro（出图，Ready）/ chartmind-ai（图表 AI，Ready）
> 设计人：验真（opc-mvp-designer）｜状态：草案，待主理人易牧确认后落盘

## 0. 本步在做什么（先说清楚 MVP 不是什么）

MVP = **最小验证（Minimum Viable Validation）**，不是「简陋产品」。

本项目的特殊前提：**产品已经 Ready**（scifigure-pro / chartmind-ai 属 19 个 Ready 产品）。所以本阶段 MVP **不是「把产品做出来」**，而是**「用最小代价验证生意假设成不成立」**——也就是验证：有没有人真的愿意付外币、收款链路能不能跑通、能不能低成本把人带来。

一人公司最优策略：先跑通「部署→收款→第一笔真付款」的最小闭环，再谈扩张。不要在还没证明有人付钱之前，就投入做功能、写文案、铺内容。

## 1. 上游假设与依赖地图

### 1.1 墨方指定 Top 5 高风险假设

| 编号 | 假设 | 当前状态标注 |
|---|---|---|
| #1 | 科研人员愿意为出图工具付费 | [假设-待确认]（付费闭环未跑，无法证伪）|
| #2 | 产品能独立运营变现（Waffo/Stripe 支付 + 部署修通）| [假设-待确认]（Waffo 待真实验证；部署脚本有硬编码绝对路径技术债）|
| #3 | 能靠 GEO + LinkedIn 低成本获客 | [假设-待确认]（GEO 引擎/LinkedIn 系统已建但效果未测；无私域）|
| #4 | 最懂人群 = 科研人员 | [假设-待确认·决定性]（分水岭；阶段02 推荐利基即建立在此）|
| #5 | 兼职 15-20h/周能持续交付 | [假设-待确认]（约束项，本阶段只监控不主验）|

### 1.2 依赖关系（谁卡谁）

```
#4 最懂人群=科研人员（决定性·分水岭）
 │
 ├─► 决定 #1 的前提：若 #4 不成立，#1「科研人员愿付」的命题本身作废
 │
#2 产品能独立变现
 │   └─[假设-高置信] 付费闭环未跑：Waffo 未真实验证、部署脚本硬编码路径技术债
 │   └─ 是 #1 的前置门控：支付跑不通 → 无法验证「愿付」
 │
#3 能低成本获客
 │   └─ 依赖 #5 兼职可持续（无私域，靠公开渠道持续发）
 │
#5 兼职可持续 ── 本阶段作为约束监控，不作为主验目标
```

**关键依赖标注：**
- [假设-高置信] 付费闭环尚未跑通过（墨方结论）。→ #2 是真实未验证项，不是「应该能行」。
- [假设-待确认] #4 是决定性分水岭：阶段 02 的利基推荐、阶段 03 的价值主张、阶段 04 的变现设计**全部建立在「最懂科研人员」之上**。一旦 #4 被推翻，下游 #1/#2/#3 的验证对象都要重锚。
- #1 与 #2 纠缠：没有工作支付，就无从测「愿付」。故 #2 是 #1 的门控。

## 2. 首验优先级决策

**结论：第一步先验证 #2（变现闭环）作为门控，并立即把 #2 与 #1 合并成一个「烟雾+真收款」实验；#3（获客）排在其后。**

理由（为什么不是先测 #3，也不是分开测 #1/#2）：

1. **#2 是物理门控。** 支付链路（Waffo/部署）不通，#1「愿付」根本没有观测手段。先把 #2 打通，成本极低（修脚本 + 接支付），却解锁了所有下游验证。
2. **#1 与 #2 共享同一个 artifact（真部署 + 真支付页）。** 分开测是重复劳动。一笔真实外币付款，同时证明「链路通（#2）」和「有人愿付（#1）」。这是性价比最高的单点验证。
3. **#3 不该排第一，但可并行做 sanity-check。** 逻辑顺序上：如果根本没人付钱（#1/#2 失败），获客效率再高也无意义。所以先 gate 在「能收到钱」，再放大「能廉价带人来」。不过，假门落地页（形态②）不依赖支付，可在 #2 打通前并行跑，提前看 LinkedIn/GEO 是否带得动点击。
4. **产品已 Ready，所以「做产品」不是风险点；「做成生意」才是。** 全部注意力应放在生意假设的证伪上。

> 可纠点：若主理人认为「先证伪 #3（获客）更便宜、风险更低」，可调整为「假门先行 + 烟雾并行」。两种排法都给出，见第 3、4 节。

## 3. 最小验证形式（Smallest MVP）—— 三方案 + 自定义

### 形态① 烟雾测试（Smoke Test）· 真部署 + 真收款
- **要建的最小 artifact**：① 修复部署脚本硬编码绝对路径技术债；② 真部署 scifigure-pro 到 Vercel 免费层；③ 接 Waffo（Stripe 为备用）真实收款，设「积分」商品；④ 一个极简「买积分」页 + 真实 checkout。
- **成本**：时间 ≈ 10–15h（修脚本 + 接支付 + 部署）；现金 ≈ $0（Vercel 免费层 + 支付费率）；AI 订阅 $50–100/月已计入预算。
- **所需时间**：1–2 周（兼职 15–20h/周）。
- **验证什么**：#2（闭环真能收到外币）+ #1（真实科研人员愿付）。
- **优点**：一次性产物，同时证伪 #2 和 #1；最贴近真实生意。
- **代价**：需修技术债、需真实接入支付（账号/合规）；需一点真实流量。

### 形态② 假门落地页（Fake-door）· 不接真支付，只测意图
- **要建的最小 artifact**：一个落地页，「买积分」按钮点击后 → waitlist 表单或「支付维护中」提示；埋点统计点击率/提交率。
- **成本**：时间 ≈ 3–5h；现金 ≈ $0。
- **所需时间**：3–5 天。
- **验证什么**：#3（LinkedIn/GEO 能否带来点击）+ #1 弱信号（愿不愿点）。
- **优点**：最快最便宜；不依赖支付打通；可在 #2 就绪前并行跑。
- **代价**：不验证 #2（真收款）；点击 ≠ 真付款，#1 信号弱；存在「假门」信任/合规风险，需明确告知。

### 形态③ 单页付费实验（Hybrid）· 真部署 + 真支付 + 极简落地 + 极小流量
- **要建的最小 artifact**：形态①的部署+支付，叠加一个转化导向单页（价值主张一句 + 价格 + 真购买按钮），用极小真实流量（已知科研人员 / 小范围 LinkedIn）跑第一笔。
- **成本**：时间 ≈ 12–18h；现金 ≈ $0。
- **所需时间**：1.5–2 周。
- **验证什么**：#2 + #1 同时验证，并顺带采样 #3 极小样本。
- **优点**：最完整的最小闭环；一笔真付款即三重信号（链路通 / 人愿付 / 渠道带得来）。
- **代价**：工作量最大；需同时备好部署与支付。

**本设计推荐：以 形态③ 作为首验主路径**（把 #2+#1 合并、顺带采样 #3）；同时以 形态② 作为不依赖支付的并行 sanity-check（提前看 LinkedIn/GEO 是否带得动点击）。

## 4. 三假设验证序列（#2 → #1 → #3）

| 步骤 | 验证目标 | 最小形态 | 退出条件（PASS）| 不通过去向 |
|---|---|---|---|---|
| **A. 门控** | #2 支付链路技术可行 | 形态①子集：接 Waffo/Stripe，自测 $1 交易 | 资金到账（3–5 工作日）| PIVOT→换 Stripe / 重构部署 |
| **B. 首验** | #1 真实科研人员愿付（含 #2 真收款）| 形态③：真部署+真支付+极小真实流量 | 见第 5 节阈值 | ITERATE 定价/价值；或触发 #4 复审 |
| **C. 放大** | #3 低成本获客（GEO+LinkedIn）| 同一已部署 artifact + GEO 引擎 + LinkedIn 自动发布 | 见第 5 节阈值 | ITERATE 内容/渠道；PIVOT 换渠道 |

衔接逻辑：
- A 是 B 的物理前提（不通则 B 无法观测）。
- B 与 A 共用同一套已部署 artifact；B 在 A 通过后，注入真实科研人员流量。
- C 复用 B 已跑通的部署+支付，只新增「放大流量 + 计量 CAC/转化」；不重复建设。
- 形态②（假门）可在 A 进行期间并行跑，作为 #3 的提前 sanity-check，不阻塞主线。

## 5. 验证方法 + 成功判据（阈值）

> 以下数值为**建议阈值，待主理人确认/纠偏**。阈值设计原则：一人公司用「第一笔真付款」作为最硬信号。

### 步骤 A — #2 门控（支付链路）
- **PASS**：自测 $1 真实交易，资金在 3–5 工作日内到达 Waffo/Stripe 账户。→ #2 技术可行，进入 B。
- **ITERATE**：交易发起但到账延迟/金额异常 → 排查支付配置/汇率/费率。
- **PIVOT**：Waffo 始终无法验证或合规受阻 → 切换 Stripe（或反之）；若两者皆不可行 → #2 被证伪，重新设计变现/分发模式（如转为 API 授权、平台内嵌）。

### 步骤 B — #1 真实付费意愿（核心）
- **初始 PASS（信号存在）**：在 ≥30 个合格科研人员访问下，收到 **≥1 笔真实外币付款**（非自测）。→ 假设 #1 拿到首个证据，进入放大。
- **强 PASS**：≥50 个合格访问 → **≥3 笔付款（≈6% 转化）**。→ #1 成立，可置信放大。
- **ITERATE**：0 真付款，但 waitlist/点击 ≥10 → 定价或价值主张问题，调整后再跑。
- **PIVOT（触发 #4 复审）**：0 真付款且 0 waitlist/0 点击 → 不是「不愿付」，可能是「找错人」。→ 进入第 6 节 #4 预案。

### 步骤 C — #3 低成本获客
- **PASS**：经 GEO+LinkedIn，月访问 **≥200**，现金 CAC ≈ $0（仅时间成本 ≤10h/月），付款转化 **≥2%**。→ #3 成立。
- **ITERATE**：有访问但 <2% 转化 → 落地/价值主张与渠道错配，调内容或页。
- **PIVOT**：30 天内容日历跑完仍近 0 访问 → GEO/LinkedIn 对该人群无效 → 换渠道（学术社区、邮件列表、KOL）。

## 6. 决定性依赖 #4 与 Pivot 预案

**#1 的决定性依赖：#4「最懂人群 = 科研人员」。** 阶段 02/03/04 全部建立于此。若 #1 在上游被推翻，**根因判定**决定走向：

- **情形 X（愿付问题，非人群问题）**：科研人员说「工具不错但太贵/不需要现在买」→ #1 属 **ITERATE**（调定价/价值/场景），#4 仍成立，不动利基。
- **情形 Y（人群错配）**：实验反复显示「科研人员不是真用户，独立开发者才是」→ #4 被推翻，触发 **PIVOT**。

### 若 #4 被推翻（其实最懂独立开发者），MVP 如何变

| 维度 | 原方案（科研人员）| Pivot 后（独立开发者）|
|---|---|---|
| 验证对象 | 科研人员愿为出图工具付外币 | 独立开发者愿为 [适配工具] 付外币 |
| 最小形态 | 同一套「真部署 + 真支付」基础设施**复用** | 同左（基础设施不变，省去重建成本）|
| 价值主张角度 | 期刊级图表 / 可复现研究 | 改为开发者向（如：API 出图、文档配图、快速原型可视化）|
| 流量来源 | LinkedIn 科研圈 / 学术 GEO | 改为 Hacker News / Product Hunt / Dev Twitter / Indie Hackers |
| 产品选择 | scifigure-pro / chartmind-ai | 从 19 个 Ready 产品中重选更贴合开发者的，或对原产品重定位 |
| 退出条件 | 同第 5 节 | 同第 5 节阈值逻辑，仅人群与渠道替换 |

**关键：基础设施（部署 + 支付）是渠道无关的**，所以 #4 被推翻时**最大沉没成本只是「流量来源 + 价值主张角度」**，不必推倒重来。这正是「先验证生意假设、产品已 Ready」的优势。

## 7. 人机分工（Human-AI Split）

| 维度 | 人工（李星亮）| AI（含已建系统）|
|---|---|---|
| 战略决策 | 决定先验哪条假设、阈值设定、Go/No-Go | — |
| 技术债修复 | 修部署脚本硬编码路径（需判断上下文）| 提供修复建议/草稿 |
| 真实支付接入 | 注册并配置 Waffo/Stripe 真实账号（合规/密钥）| 生成配置步骤清单 |
| 内容/获客 | 定性判断渠道有效性、与科研人员对话拿定性信号 | GEO 内容引擎生成、LinkedIn 自动发布（已建）|
| 数据与判定 | 解读结果、决定 pivot/iterate | 埋点、看板、A/B 文案变体生成 |
| 问责 | 对「是否继续投入」最终负责 | 执行可规模化、可重复的部分 |

原则：AI 承接可重复、可规模化的执行；人工守住判断与问责（一人公司的「人」不可外包）。

## 8. MVP 边界（含越界提示）

**本阶段 IN（做）：**
- 锁定首验假设（#2 门控 + #1 主验 + #3 放大）
- 选定最小验证形态（烟雾/假门/单页付费）
- 明确成功判据与退出条件
- 明确人机分工

**本阶段 OUT（不做，留给后续阶段）：**
- ❌ 具体广告文案、LinkedIn 帖子骨架、落地页正文 → 阶段 07 转化闭环（opc-conversion-loop）
- ❌ 完整转化路径 / 漏斗设计 → 阶段 07
- ❌ 交付 SOP、运营细节 → 阶段 08+ 资产沉淀（opc-asset-ops）
- ❌ 产品功能开发 → 产品已 Ready，本阶段不新增功能

> 越界检测：若出现「第一篇帖子写什么」「落地页正文怎么写」「具体怎么交付」等执行话题，先记录，留待转化闭环与资产沉淀阶段。**先把验证路径框架确认好。**

2. **#1 决定性依赖 #4**：是否认可「阶段 02/03/04 全部建立在科研人员之上」，并预设好 #4 被推翻时的 Pivot 预案？
3. **成功阈值**：第 5 节的数值（30 访问/1 付款、50 访问/3 付款、200 访问/2% 转化）是否合适？需按真实渠道体量调整。
